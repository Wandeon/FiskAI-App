## Design Philosophy

### Three Pillars

**1. Like a Game, Not an ERP**

Traditional ERP software shows 50 tabs, endless tables, and treats every user the same. FiskAI should feel like a well-designed game:

- **Progressive disclosure** - reveal complexity only when needed
- **Contextual actions** - options appear where/when relevant
- **Depth on demand** - simple surface, drill down when curious
- **Mastery curve** - grows with the user
- **Satisfying interactions** - instant feedback, smooth animations

**2. Zero Redundant Input (with Accountability)**

Never ask for information we already have or can derive, but always show provenance:

```
Data Intelligence Hierarchy:

1. AUTOMATIC - System knows â†’ shows it with source + confidence
   â€¢ Invoice paid (from bank statement, confidence: HIGH)
   â€¢ Client address (from registry lookup, confidence: MEDIUM)
   â€¢ Next invoice number (from sequence, confidence: HIGH)

   â†’ Every auto-filled field shows: source, confidence, "Edit" option

2. SUGGESTED - System guesses â†’ user confirms with one click
   â€¢ "This payment matches invoice #002" [Confirm / Wrong]
   â€¢ "Same service as last time?" [Yes / No]

   â†’ Shows match reason, allows override

3. ASSISTED - System helps â†’ user picks from options
   â€¢ "3 possible matches - which one?" [A / B / C / None]

   â†’ Presents options with confidence scores

4. MANUAL (last resort) - System can't help â†’ user types
   â€¢ Custom service description
   â€¢ Cash payment (no bank record)
   â€¢ Foreign client (no Croatian OIB)

   â†’ Clear "we don't have this data" messaging
```

**3. Visual Precision**

Every element sized to its content, not to a grid:

- Date fields are date-sized, not full-width
- Group fields by meaning (WHO / WHEN / WHAT)
- Whitespace creates hierarchy
- Mobile-first design that enhances on desktop

## Compliance State Model

### States

Every company has a deterministic compliance state, evaluated continuously:

```typescript
enum ComplianceState {
  OK = "OK", // All obligations met, no upcoming risks
  ATTENTION = "ATTENTION", // Action needed within 14 days
  RISK = "RISK", // Overdue or critical issue
}

interface ComplianceStatus {
  state: ComplianceState
  evaluatedAt: DateTime
  reasons: ComplianceReason[]
  nextEvaluation: DateTime
}

interface ComplianceReason {
  code: string // e.g., 'DEADLINE_APPROACHING', 'MISSING_SETUP'
  severity: "info" | "warning" | "critical"
  message: string // Human-readable
  action: string // What to do
  actionUrl: string // Where to do it
  dueDate?: DateTime // If deadline-related
}
```
